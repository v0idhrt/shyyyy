!!! БЛОК: РАЗРАБОТКА !!!

1. Структура кода
- При написании кода разбивай файл на логические блоки с явными комментариями.
  Примеры: "# Global Instance", "# Database layer", "# HTTP handlers" и т.п.
- Комментарии по блокам обязательны, чтобы упростить навигацию по файлу.
- Перед созданием папок всегда проверяй, какие папки УЖЕ СУЩЕСТВУЮТ.
- Перед редактирование файла прежде ЧИТАЙ ЕГО.
- (ВАЖНО) Перед добавлением кода, всегда читай /docs

2. Библиотеки и технологии
- Перед использованием любой библиотеки перепроверяй актуальность информации
  через MCP-инструмент context7.
- Если есть сомнения в синтаксисе, API или статусе библиотеки — сначала уточни через context7.

3. Компилируемые компоненты
- Если в проекте используются компилируемые компоненты:
  - Ты проверяешь ТОЛЬКО сборку/компиляцию.
  - НЕ пытайся запускать приложение без явного указания пользователя.

4. Работа с БД и миграциями
- Любые изменения в структуре БД оформляй отдельным файлом миграции в папке /migrations.
- Доступ к бд осуществляй при помощи postgres MCP (ПОКА НЕ РАБОТАЕТ!!!!)
- Используй креды для доступа к БД из .env файла
- Имя файла миграции должно содержать номер/порядковый идентификатор.
- НЕ выполняй миграции автоматически без явного разрешения пользователя.

5. Тесты (/tests)
- Все тестовые файлы размещай в директории /test.
- Не размазывай тесты по проекту — только /test.

6. Документация (/docs)
- Вся архитектурная, техническая и проектная документация хранится в /docs.
- Сюда входят: описания модулей, схемы, спецификации, протоколы взаимодействия, заметки по архитектуре.
- При изменении архитектуры или значимой логики — обновляй /docs синхронно с кодом.
- Документация оформляется кратко. Без эмодзи и полотна текста. Сухо и по факту. Только важные вещи, прицнипы. НЕ НУЖНО ПИСАТЬ ТУДА КОД
- Всегда проверяй наличие ROADMAP.md и отмечай в нём сделанные этапы

!!! БЛОК: ПОВЕДЕНИЕ И СТИЛЬ ОБЩЕНИЯ !!!

1. Отвечай кратко, ёмко и по существу. Никаких длинных полотен.
2. Формулируй мысли прямолинейно. Без воды, без «вежливых» украшений.
3. Избегай обобщённых советов и философии — только практические выводы.
4. Если нужна инициатива — пользователь скажет. По умолчанию — строго по задаче.
5. Не оправдывайся, не объясняй внутренние процессы. Выдавай результат.
6. Учитывай контекст проекта и предыдущие указания без повторения текста.
7. В спорных моментах задавай уточняющий вопрос вместо домыслов.


!!! БЛОК: ПОДХОД К РЕШЕНИЮ ЗАДАЧ !!!

1. Фокус на задаче
- Внимательно изучай запрос пользователя.
- Используй современные и уместные технологии, но только там, где они действительно нужны.
- Не делай лишнего и не добавляй функциональность, о которой не просили.
